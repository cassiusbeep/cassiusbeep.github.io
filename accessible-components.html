accessible components: a case study

search bars are everywhere, and before this study i thought they were pretty much the same everywhere. in this study i consider the design of this component across different platforms and use what i find to design a new version of the firefox search bar.

part 1: research back to top

a search bar's main job is to take user input. let's take a look at how different bars change in response to our actions.

firefox's search bar moves through 4 different states in response to user input:

initial,

auto-focus,

focused,

... and active

citymapper, a mobile and web navigation app, has a search bar for the start location and the destination. each has 3 states:

initial,

focused,

... and active.

windows 11's file explorer has 4 states:

initial,

hover,

focused,

... and active.

i wanted to know how these different implementations stack up in terms of interaction, functionality, and accessibility. i found that some components were designed better than others when it came to discoverability and learnability. for example:

  • LIKE: firefox has a button to delete search suggestions, which appears on hover. it's intuitive and learnable because of the wide area that triggers it.
  • DISLIKE: firefox's system of toggling between switch to tab/create new tab modes. it's not discoverable at all! I assumed clicking the "switch to tab" label would switch, while clicking elsewhere would create a new tab. instead, clicking anywhere will switch, and you have to hold specific keys to create a new tab.
  • LIKE: both Citymapper and Explorer use a delete button which appears only when the user starts to type. this balances minimalism with discoverability
  • DISLIKE: the interactivity of symbols inside the search bar is inconsistent across each app. every time, i hope i can click the magnifying glass to submit.

my next goal is to look at the output of components. how are they experienced by the user - and how does this change based on their unique perception? for example, if they perceive visual output, colour, or use audio i.e. with a screen reader. to link the input and output of components, i created state models describing how the firefox search bar changes as you use it.

part 2: redesign back to top

as an exercise in experience design and meaningful tradeoffs, I tried to come up with a design change for the mouse user state model that would provide some benefit to mouse users but a deliberate downside (in memorability, learnability, accessibility, or efficiency) for keyboard users.

I separated the "Switch to Tab" element from the main element in the suggestion list, creating a separate button so that two different functions are separate targets instead of a toggleable mode. this means that the default behaviour of clicking a suggested item in the list will create a new tab with that content, and only the "Switch to Tab" button itself will perform the tab switching behaviour.

now keyboard users have double the elements to tab through when navigating the list, but mouse users have more clarity in what their click will do.

my redesign for the keyboard user experience removes the auto-focused state of the search bar. this means that when a new tab is opened or the search bar is tabbed into, the list automatically pops up with suggestions for keyboard users to browse.

this auto-focus would likely annoy mouse users, as it gets in the way of the page they're trying to click on. but it makes navigation faster for keyboard users.

to mitigate the downside, the browser could close the list as soon as the user moves focus from the browser header (when the cursor moves down to the page).

The diagram also reflects the changes to the focus order made by my mouse state model redesign, and sacrifices the redundancy between the arrow and tab keys to create a quick/efficient way for keyboard users to exit the suggestion list and move to other components.

with these changes in mind, I sketched a possible redesign for the Firefox search bar component that incorporate my critiques above. the first shows the initial state upon opening a new tab, and the second is the state when a keyboard user tabs onto the "Switch to tab" button on a list element. the annotations in the diagram explain more of my thoughts behind the redesign process.

i decided to make tab duplication rather than tab switching the "default" behaviour. I think that the majority of web browser users have a low number of webpages open, and tend to navigate purely by Tabbing/clicking between the visual handles at the top of the page.

therefore, most people won't use the existing search bar workflow to find an existing tab. I think if they perform the exact same action as they did to open the first tab, the same behaviour should occur.

It makes more sense for those expecting a different result to learn one more step (clicking or tabbing to the button) than changing their behaviour to achieve the same result (opening a new tab).

overall, this case study gave me an excuse to dive deeper in one of the most widely used components we have, and think more deliberately about how i can implement it in future.
